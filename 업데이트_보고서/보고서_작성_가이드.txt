[2] 버전 관리 규칙
================================================================================

Semantic Versioning (유의적 버전):

  v [메이저] . [마이너] . [패치]
  
  메이저 (첫번째 번호):
    - 사용자 인터페이스 대폭 변경
    - 하위 호환성 완전 깨짐
    - 예: v1.0 → v2.0
  
  마이너 (두번째 번호):
    - 새로운 기능 추가
    - 기능 개선
    - 하위 호환성 유지
    - 예: v2.3 → v2.4
  
  패치 (세번째 번호):
    - 버그 수정
    - 소규모 개선
    - 하위 호환성 유지
    - 예: v2.3.1 → v2.3.2


버전 변경 예시:

  현재: v2.3 (기능 개선 완료)
  버그 수정 1회: v2.3.1
  버그 수정 2회: v2.3.2
  새 기능 추가: v2.4
  새 기능에서 버그: v2.4.1
  
  대규모 리팩토링: v3.0

4.1 버그 수정 보고서 구조
─────────────────────────────────────────────────────────────────────────────

필수 섹션:
  [버그 설명] - 어떤 문제가 발생했는가
    → 증상, 근본 원인, 영향 범위
  
  [해결 방법] - 어떻게 해결했는가
    → 수정된 로직, 코드 변경사항
  
  [테스트 방법] - 어떻게 검증하는가
    → 테스트 절차, 예상 결과
  
  [호환성] - 기존 코드와 문제없는가
    → 라이브러리, 하위 호환성
  
  [성능 영향] - 성능이 어떻게 변했는가
    → 속도, 메모리, 리소스 영향
  
  [결론] - 최종 평가
    → 수정이 완전한지, 부작용은 없는지


4.2 기능 개선 보고서 구조
─────────────────────────────────────────────────────────────────────────────

필수 섹션:
  [개선 배경] - 왜 개선하는가
    → 현재 문제점, 개선 목표
  
  [개선 내용] - 무엇을 개선했는가
    → 구체적인 변경사항, 파라미터 변화
  
  [기대 효과] - 어떤 이점이 있는가
    → 정량적 효과 (성능 향상 %)
    → 정성적 효과 (사용성 개선)
  
  [사용 방법] - 어떻게 사용하는가
    → 기본 사용법, 고급 옵션
  
  [주의사항] - 무엇을 조심해야 하는가
    → 성능 주의사항, 비용 영향
  
  [버전 비교] - 이전 버전과 비교
    → 기능 비교 표, 개선 정도


4.3 새 기능 추가 보고서 구조
─────────────────────────────────────────────────────────────────────────────

필수 섹션:
  [기능 개요] - 새로운 기능이 무엇인가
    → 목적, 사용 사례
  
  [기술 사양] - 기술적으로 어떻게 구현됐나
    → 알고리즘, 아키텍처, 의존성
  
  [사용 가이드] - 어떻게 사용하는가
    → 단계별 사용 방법, 예시
  
  [설정 옵션] - 어떤 설정이 가능한가
    → 파라미터, 기본값, 권장값
  
  [성능 및 제약] - 어떤 한계가 있는가
    → 처리 속도, 리소스 요구사항
  
  [향후 개선] - 앞으로 개선할 사항
    → 기능 확장, 최적화 계획

5.1 언어 및 톤
─────────────────────────────────────────────────────────────────────────────

✓ 사용할 것:
  - 존경어 (존댓말 X)
  - 간결한 문체
  - 구체적인 수치
  - 명확한 구조

✗ 사용하지 말 것:
  - 존댓말 ("~해", "~해야 해")
  - 모호한 표현 ("대략", "대충")
  - 문학적 표현 ("아름다운", "멋진")


5.2 구조 및 레이아웃
─────────────────────────────────────────────────────────────────────────────

구분 문자:
  메인 제목: ================================================================================
  부제목:    [내용]
  소제목:    ─────────────────────────────────────────────────────────────────────────────

리스트:
  주요 항목: ✓ 항목
  주의사항:  ⚠️ 항목
  효과:      ↑ 항목

코드:
  ```
  코드 내용
  ```

테이블:
  파라미터              이전값      개선값      이유
  ──────────────────────────────────────────────────────
  예시                  10          20        설명


5.3 시각적 요소
─────────────────────────────────────────────────────────────────────────────

이모지 사용:
  ✅ 성공, 완료, 지원함
  ❌ 실패, 미지원, 없음
  ⚠️ 주의, 경고
  💡 팁, 아이디어
  🔧 도구, 기술적
  📊 통계, 데이터
  🚀 시작, 시작 가이드
  📌 중요
  ↑ 개선, 향상
  ↓ 저하, 감소
  ✓ 체크, 긍정
  ✗ 미지원, 부정

[9] 우수 사례 (Best Practices)
================================================================================

DO:
  ✓ 변경 사항 완료 직후 보고서 작성
  ✓ 구체적인 테스트 절차 기술
  ✓ 사용자 관점에서 작성
  ✓ 이전 버전과 명확히 비교
  ✓ 호환성 상태 명확히 표시
  ✓ 향후 계획 언급
  ✓ 코드 스니펫 포함

DON'T:
  ✗ 너무 길게 작성 (1페이지 초과 지양)
  ✗ 기술 용어 남용
  ✗ 모호한 표현
  ✗ 숫자 근거 없이 주장
  ✗ 호환성 명시 누락
  ✗ 파일명 규칙 무시
